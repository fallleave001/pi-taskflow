---

**الوكلاء الفرعيون (subagents) هم إطلاق وانتهاء. الـ Taskflows تُطلق، تتوسع، تتوقف، تُحكم، تستأنف، وتحفظ نفسها كأمر.**

أنت تعرف بالفعل أدوات `task` / `tasks` / `chain` المضمنة للوكلاء الفرعيين. `pi-taskflow` يتحدث نفس الاختصارات — لذا فإن تفويضاتك الحالية تصبح فوراً **متتبعة، قابلة للاستئناف، وقابلة للحفظ كأمر بكلمة واحدة `/tf:<name>`**. عندما تتجاوز الاختصارات، تمنحك DSL الكاملة DAG حقيقي: توسيع ديناميكي على عشرات العناصر، توجيه شرطي، بوابات جودة، موافقات بشرية، إعادة محاولة، وسقف إنفاق صارم.

وكل هذا الوقت، **فقط المرحلة النهائية تصل إلى محادثتك.** كل transcript وسيط يبقى في بيئة التشغيل، وليس في نافذة السياق الخاصة بك.

## لماذا هذا موجود

هذا هو الجدار الذي تصطدم به مع الوكلاء الفرعيين الخام: تصف خطة متعددة الخطوات نثراً، والنموذج يعيد استنباطها في كل مرة تشغيل، وتُغرق transcripts الوسيطة سياقك، وعند فشل استدعاء نموذج واحد تبدأ من الصفر. لا إعادة استخدام، لا استرداد، لا هيكل.

`pi-taskflow` ينقل الخطة **خارج الـ prompt وإلى تعريف تعريفي.** بيئة التشغيل تمتلك الـ DAG، والحلقات، وإعادة المحاولة، والحالة الوسيطة. أنت تعلن عن pipeline مرة واحدة وتشغلها مئة مرة — بالاسم.

> عندما تحتاج وظيفة ما إلى اثنتي عشرة خطوة مع توسيع متفرع وبوابة مراجعة، فأنت بحاجة إلى تنسيق (orchestration) — وليس حظاً في الـ prompting.

| | الوكيل الفرعي (subagent — مدمج) | **pi-taskflow** |
|---|---|---|
| **من يقود** | النموذج، خطوة بخطوة | بيئة التشغيل، من تعريف |
| **الطوبولوجيا (Topology)** | سلسلة / متوازٍ مسطح | **DAG مع تزامن طبقي + توجيه (routing)** |
| **النتائج الوسيطة** | في نافذة السياق الخاصة بك | **في بيئة التشغيل — وليس في سياقك** |
| **النطاق (Scale)** | حفنة من المهام | **توسيع `map` ديناميكي على عشرات العناصر** |
| **قابلية إعادة الاستخدام** | يُعاد وصفه كل مرة | **يُحفظ كـ `/tf:<name>`** |
| **قابلية الاستئناف (Resumable)** | ✗ | **✓ عبر الجلسات — المراحل المخبأة تُتخطى تلقائياً** |
| **بوابات الجودة (Quality gates)** | ✗ | **مراحل `gate` تتوقف عند `VERDICT: BLOCK`** |
| **التوجيه الشرطي (Conditional routing)** | ✗ | **حراس `when` + روابط `join: any` OR** |
| **تحمل الأخطاء (Fault tolerance)** | ✗ | **إعادة محاولة `retry` لكل مرحلة + إعادة تلقائية للأخطاء العابرة** |
| **الإنسان في الحلقة (Human-in-the-loop)** | ✗ | **مراحل `approval` (موافقة / رفض / تعديل)** |
| **التحكم في التكلفة (Cost control)** | ✗ | **ميزانية `budget` للمدى الكامل (سقف USD / رمز مميز)** |
| **التركيب (Composition)** | ✗ | **مراحل `flow` تشغّل تدفقات فرعية محفوظة** |
| **التقدم المباشر (Live progress)** | غير شفاف أثناء التشغيل | **عرض DAG حي مع التوقيت والتكلفة** |
| **بيئة العمل (Ergonomics)** | JSON مضمن كل مرة | **اختصار (`task`/`tasks`/`chain`) *أو* DSL** |

إنه لا يستبدل أداة الوكيل الفرعي. إنه يمنح وكلاءك الفرعيين DAG، وذاكرة، واسماً.

**كيف تختار:**

- **`@pi-agents/orchid`** هو أكثر مُنسّق اكتمالاً في الميزات في النظام البيئي (DAG + worktrees + حلقة Ralph + صندوق بريد الوكيل) — لكن DSL الخاص به هو pipeline *ثابت* من 9 مراحل، ويحمل تبعيات وقت تشغيل + jiti، وهو في مرحلة beta. استخدم `pi-taskflow` عندما تريد **تعريف الرسم البياني الخاص بك** (وليس تبني رسم بياني رأيي) مع **صفر تبعيات** وتثبيت بأمر واحد.
- **`pi-crew` / `ultimate-pi`** يذهبان إلى مستوى أثقل — عزل worktree، فرق غير متزامنة متينة، حوكمة متعددة المستويات. إذا كنت تريد خفيفاً، تعريفياً، وبدون تبعيات، فهذا هو المشروع المناسب.
- **`@zhushanwen/pi-workflow`** هو الأقرب روحياً وأيضاً بدون تبعيات، لكنك تؤلف التدفقات العمل كـ **نصوص JavaScript**. DSL الـ JSON التعريفي الخاص بـ `pi-taskflow` أكثر أماناً وقابلية للتدقيق، واستئنافه على مستوى بصمة إدخال المرحلة أكثر دقة من إلغاء التكرار بواسطة ذاكرة تخزين الاستدعاءات.
- **`@fiale-plus/pi-rogue-orchestration`** لديه **حلقة حتى الاكتمال** حقيقية (ميزة لا يمتلكها `pi-taskflow` بعد). إذا كانت وظيفتك هي "استمر حتى يتحقق الهدف"، فهي تستحق النظر؛ `pi-taskflow` مصمم لـ *pipeline منظمة متفرعة* بدلاً من ذلك.
- **`pi-subagents` / `@gotgenes/pi-subagents`** هما الخياران الناضجان للتفويضات المخصصة "استخدم المراجع على هذا الفرق" والوظائف الخلفية. `pi-taskflow` مخصص عندما تحتاج هذه التفويضات أن تصبح *pipeline قابل للتكرار والاستئناف*.
- **`pi-pipeline` / `pi-agent-flow`** تقدم تدفقات *ثابتة رأيية*. `pi-taskflow` يقدم *لوحة فارغة*: أنت (أو النموذج) تعلن الرسم البياني الذي يناسب المهمة.

> الجملة الصادقة المختصرة: **`pi-taskflow` هي إضافة Pi الوحيدة التي تمنحك pipeline وكيل فرعي DAG تعريفية قابلة للاستئناف وتحفظ كأمر بكلمة واحدة — مع صفر تبعيات وقت تشغيل وعزل السياق حسب التصميم.** الفجوات المعروفة التي سيتم سدها قريباً: حلقة حتى الاكتمال، عزل worktree، وتشغيلات خلفية غير محظورة (راجع [`STRATEGY.md`](./STRATEGY.md)).

## بداية 30 ثانية

**1. التثبيت** — أمر واحد:

```bash
pi install npm:pi-taskflow
```

> **اختياري:** شغّل `/tf init` مرة واحدة لربط أدوار النموذج للوكلاء الـ 18 المدمجين (`fast`، `strong`، `thinker`، …) بنماذجك الخاصة — منتقي تفاعلي. تخطّه وسيستخدم الوكلاء نموذج Pi الافتراضي فقط. راجع [أدوار النموذج](#model-roles).

**2. التشغيل** — فقط اطلب من النموذج في جلسة Pi:

> *شغّل سلسلة: أولاً استكشف تدفق المصادقة، ثم لخص النتائج.*

يقوم النموذج باستدعاء أداة `taskflow` تلقائياً. ستحصل على تقدم مباشر، توقيت لكل خطوة، تكلفة الرمز المميز، وسجل تشغيل محفوظ — **نفس الجهد المبذول مع الأداة المدمجة، لكنه الآن متتبع وقابل للاستئناف.**

**3. الحفظ** — قل *"احفظه"* وسيكون لديك `/tf:<name>` إلى الأبد.

هذا كل شيء. يمكنك تشغيل أول تدفق عمل لك قبل أن يبرد قهوتك — دون كتابة تعريف مرحلة واحدة.

> 📖 التفاصيل الكاملة (الاختصارات، أنواع المراحل، DSL، التوجيه الشرطي، إعادة المحاولة، الموافقات، الحلقات، البطولات، التخزين المؤقت عبر التشغيلات، الاستئناف عبر الجلسات، اللوحات التفاعلية، إعدادات أدوار النموذج، الوكلاء المدمجون، خريطة النظام البيئي، والأمثلة) — راجع [English README](../README.md).

على نفس الشكل الذي اعتدت عليه من `task` / `tasks` / `chain`:

```jsonc
// Single — وكيل واحد، مهمة واحدة
{ "task": "لخص بنية src/", "agent": "explorer" }

// Parallel — أطلق عدة دفعة واحدة، المخرجات تندمج
{ "tasks": [
  { "task": "دقق المصادقة في src/api",             "agent": "analyst" },
  { "task": "دقق التحقق من الإدخال في src/api", "agent": "analyst" }
] }

// Chain — متسلسل؛ كل خطوة ترى مخرجات السابقة
{ "chain": [
  { "task": "اسرد الـ API العام لـ src/lib", "agent": "scout" },
  { "task": "اكتب توثيقاً لـ:\n{previous.output}", "agent": "writer" }
] }
```

`agent` اختياري (الافتراضي هو أول وكيل مكتشف). أضف `name` لتسمية التشغيل وفتح إمكانية حفظه كأمر.

## شاهده أثناء التشغيل

هذه ليست محاكاة. **هذا هو stdout من تشغيل حقيقي** — تدفق `self-improve` الذي يكتب ويتحقق من مجموعات الاختبار الخاصة به، تم اعتراضه في منتصف الرحلة بواسطة بوابة جودة:

```
⊗ taskflow self-improve  6/7 · blocked · $0.095
    ✓ discover            agent   deepseek-v4-flash  10t ↑38k ↓6.7k $0.011
  ┌ ✓ write-runner-tests  agent   claude-sonnet-4-6  10t ↑13 ↓6.6k $0.020
  ├ ✓ write-store-tests   agent   claude-sonnet-4-6  10t ↑11 ↓10k $0.018
  ├ ✓ write-agents-tests  agent   claude-sonnet-4-6  10t ↑28 ↓13k $0.030
  └ ✓ fix-stability       agent   claude-sonnet-4-6  10t ↑13 ↓3.9k $0.012
    ✓ verify              gate    BLOCK 3 type errors in test files  deepseek-v4-flash
    ⊘ report              reduce  skipped · Gate blocked  ↳ fix-stability
```

**التخطيط *هو* الـ DAG.** لا لوحة تحكم، لا سجلات للبحث — تقرأ شريط التقدم وتفهم الـ pipeline بأكمله:

- **الرأس** — `⊗` = محظور (بوابة أوقفته)؛ `6/7` مراحل معالجة؛ التكلفة الإجمالية `$0.095`.
- **أيقونات الحالة** — `✓` تم · `◐` قيد التشغيل · `✗` فشل · `⊘` تم تخطيه · `○` معلق.
- **السكة `┌ ├ └`** — مراحل في نفس طبقة DAG، تعمل بشكل متزامن. المهام الأربع `write-*`/`fix-stability` تتفرع من `discover`. حضيض فارغ = طبقة أحادية المرحلة.
- **`↳`** — تبعية طويلة تتجاوز الطبقات. `report` تعتمد على `verify` المجاورة *وعلى* `fix-stability` على بعد طبقتين، لذا فقط حافة التخطي مشروحة.
- **البوابة** — أصدرت `verify` `VERDICT: BLOCK`، لذا تخطت بيئة التشغيل `report` وأنهت التشغيل كـ `blocked`، مع عرض السبب في السطر.
- **التفاصيل** — لكل مرحلة: النموذج، أعداد الرموز المميزة (`↑` واردة `↓` صادرة)، التكلفة، التوقيت. مراحل التوسع تظهر أيضاً تقدم المهام الفرعية (`3/15 2✗ 8▸`).

## الانتقال إلى التعريفية

الاختصار هو منحدر الدخول. DSL هي حيث تثبت `pi-taskflow` جدارتها — توسع ديناميكي، توجيه منظم، وبوابات جودة.

> 📖 التوثيق الكامل للـ DSL (أنواع المراحل، الحقول، `when`، `join: any`، `retry`، `approval`، `flow`، `loop`، `tournament`، `cache`، `budget`، التعبيرات، `map`/`reduce`، أمثلة كاملة) — راجع [English README](../README.md).

- **صفر تبعيات وقت تشغيل.** لا حقل `dependencies` — بيئة التشغيل مبنية بالكامل على أدوات Node المدمجة (`fs` / `path` / `os` / `child_process` / `crypto`). قفل الملف هو `fs.openSync("wx")`، وليس مكتبة طرف ثالث.
- **394 اختباراً عبر 14 مجموعة** تغطي التزامن، قفل الملفات الذري (تراجعات سباق 8 عمليات)، تحصين اختراق المسار، الاستئناف عبر الجلسات، نضارة التخزين المؤقت عبر التشغيلات (عزل مفتاح التدفق/التفكير/الأدوات، إبطال البصمة، إخلاء TTL/LRU)، أحكام البوابة، حدود الميزانية، إعادة المحاولة/التراجع، تدفقات الموافقة، إنهاء الحلقة، تحكيم البطولة، تركيب التدفق الفرعي، عزل رد النداء، المراقب الخامل، تكوين دور النموذج التهيئي، و parseModelFromLabel مع تراجع اسم النموذج المقوس — بالإضافة إلى اختبار شامل من البداية إلى النهاية يشغل وكلاء فرعيين حقيقيين واستهلاك ذاتي للتخزين المؤقت عبر التشغيلات.
- **محصن بالتصميم.** دفاع ضد اختراق المسار (معجمي + `realpath`)، التحقق من runId، تعقيم HTML/الأخطاء، كتابات ذرية، سرقة القفل العالق عبر `rename`، ومراقب خامل يقتل الوكلاء الفرعيين العالقين.
- **استهلاك ذاتي (Dogfooded).** كل ميزة جديدة يجب أن تنجو من تدفق `self-improve` الخاص بالمشروع قبل أن تُشحن.

## 🍽️ نحن نأكل من طعامنا بأنفسنا

كل ميزة في `pi-taskflow` تُشحن **عبر `pi-taskflow`.**

تدفق `self-improve` الخاص بنا هو DAG من 10 مراحل — يدقق قاعدة البيانات، يصحح العيوب، يتحقق من الصحة، يحكم على الجودة، ويعرض التقرير — كل ذلك تعريفياً. تم حفظه كـ `/tf:self-improve` ويُشغل قبل كل إصدار. لا يوجد مُنسّق وكلاء آخر في النظام البيئي لـ Pi يبني نفسه بنفسه.
